# App Specification: Todo

## Overview

A to-do list manager that synchronises items between a REST API server and local
storage. The app works fully offline -- queuing mutations and flushing them when
connectivity is available -- and receives real-time push updates from the server
so changes made by other clients appear automatically.

## Features

- **View items** -- the app displays all to-do items from the local store on launch.
- **Add item** -- user enters a title and taps "Add"; a new item appears immediately
  with a client-generated CUID v2 identifier.
- **Edit title** -- user edits an existing item's title; the change is saved locally
  and pushed to the server.
- **Toggle completion** -- user taps an item to toggle it between active and completed.
- **Delete item** -- user removes an item from the list.
- **Clear completed** -- user taps a "Clear completed" button; all completed items
  are removed from the list and a delete operation is queued for each removed item.
- **Offline operation** -- all of the above work without network connectivity.
  Mutations are queued locally and flushed when the server becomes reachable.
- **Automatic sync** -- pending operations are sent to the server immediately when
  possible. If the request fails, operations remain queued and are retried on a
  periodic timer.
- **Real-time updates** -- the app subscribes to a server-sent event stream. When
  another client creates, updates, or deletes an item, the change is applied
  locally after conflict resolution.

## Data Model

- A list of **to-do items**, each with:
  - `id` -- a CUID v2 string, generated by the client at creation time.
  - `title` -- non-empty text.
  - `completed` -- boolean flag.
  - `updated_at` -- UTC timestamp of the most recent mutation (used for
    last-writer-wins conflict resolution).
- A **pending operation queue** -- an ordered list of operations that have not yet
  been confirmed by the server. Each operation is one of:
  - `Create(TodoItem)` -- a new item to POST.
  - `Update(TodoItem)` -- a changed item to PUT.
  - `Delete(id)` -- an item ID to DELETE.
- A **filter** selection: `All`, `Active`, or `Completed`.
- A **sync status** flag: `Idle`, `Syncing`, or `Offline`.
- A **last synced at** timestamp (optional, for display purposes).
- An **SSE connection state**: `Connected`, `Disconnected`, or `Connecting`.

## User Interface

- A text input field and an "Add" button at the top of the screen.
- A scrollable list of items. Each item shows:
  - A checkbox reflecting the `completed` state.
  - The item title (with strikethrough styling when completed).
  - A delete button or swipe action.
- A footer showing:
  - Count of remaining active items (e.g. "3 items left").
  - Sync status indicator: a green dot when synced, an amber dot with pending
    count when changes are queued, a red dot when offline.
  - A "Clear completed" button, visible only when at least one completed item
    exists.
- Filter tabs: **All**, **Active**, **Completed**. The selected filter controls
  which items appear in the list.

## Capabilities

| Capability | Needed? | Details |
|---|---|---|
| **HTTP** | Yes | CRUD operations against the REST API (see API Details). |
| **Key-Value storage** | Yes | Persists the item list and pending operation queue locally so the app survives restarts and works offline. |
| **Timer / Time** | Yes | Periodic retry timer (e.g. every 30 seconds) to flush the pending queue when previous attempts failed. Also provides `now()` for `updated_at` timestamps. |
| **Server-Sent Events** | Yes | Subscribes to a push stream from the server to receive create/update/delete notifications from other clients. |
| **Platform detection** | No | |

## API Details

### GET /api/todos

Returns all to-do items.

```json
[
  {
    "id": "cm1a2b3c4d5e6f7g8h9i0",
    "title": "Buy milk",
    "completed": false,
    "updated_at": "2025-06-15T10:30:00Z"
  }
]
```

### POST /api/todos

Creates a new to-do item. The client sends a complete item including its
client-generated ID.

Request:

```json
{
  "id": "cm1a2b3c4d5e6f7g8h9i0",
  "title": "Buy milk",
  "completed": false
}
```

Response (201 Created):

```json
{
  "id": "cm1a2b3c4d5e6f7g8h9i0",
  "title": "Buy milk",
  "completed": false,
  "updated_at": "2025-06-15T10:30:00Z"
}
```

### PUT /api/todos/{id}

Updates an existing item. The client sends the changed fields.

Request:

```json
{
  "title": "Buy oat milk",
  "completed": true
}
```

Response (200 OK):

```json
{
  "id": "cm1a2b3c4d5e6f7g8h9i0",
  "title": "Buy oat milk",
  "completed": true,
  "updated_at": "2025-06-15T10:31:00Z"
}
```

### DELETE /api/todos/{id}

Deletes a to-do item.

Response: `204 No Content`

### GET /api/todos/events (SSE)

A server-sent event stream. The server emits events whenever any client mutates
to-do data.

Event types:

- `item_created` -- data payload is a full `TodoItem` JSON object.
- `item_updated` -- data payload is a full `TodoItem` JSON object.
- `item_deleted` -- data payload is `{ "id": "..." }`.

Example:

```
event: item_updated
data: {"id":"cm1a2b3c4d5e6f7g8h9i0","title":"Buy oat milk","completed":true,"updated_at":"2025-06-15T10:31:00Z"}
```

## Business Rules

- **Non-empty titles** -- item titles must not be empty or consist only of
  whitespace. The "Add" action is a no-op if the input is blank.
- **Client-generated IDs** -- the client generates a CUID v2 for each new item.
  The server accepts and preserves client-supplied IDs.
- **Last-writer-wins** -- when a server push delivers an update for an item that
  also has a pending local mutation, the version with the later `updated_at`
  timestamp wins. If timestamps are equal, the server version wins (tie-break
  in favour of the shared truth).
- **Queue flush with skip** -- pending operations are flushed to the server in
  the order they were created. If an individual operation fails, it is skipped
  and the flush continues with the remaining operations. Skipped operations
  remain in the queue for the next retry cycle. (NOTE: this skip-and-continue
  strategy may need revisiting -- skipping a failed Create and then processing
  a subsequent Update for the same item will also fail. A future iteration
  should consider dependency-aware skip logic.)
- **Retry on timer** -- when an immediate sync attempt fails (network error or
  server 5xx), the queue is retried after a fixed interval (30 seconds).
- **Idempotent creates** -- because the client supplies the ID, a retried POST
  for an item that already exists on the server may receive a `409 Conflict`.
  The client handles this by converting the Create operation into a PUT
  (update) and retrying immediately.
- **SSE reconnection** -- if the SSE connection drops, the app attempts to
  reconnect. On reconnect it fetches the full item list (GET /api/todos) to
  catch up on any events missed during the disconnect. (NOTE: refetching the
  entire list on every reconnect is simple but potentially expensive at scale.
  A future improvement could use `Last-Event-ID` to replay only missed events.)
- **Clear completed** -- removes all completed items atomically. A Delete
  pending operation is queued for each removed item.
- **Local persistence** -- the full item list and pending queue are written to
  Key-Value storage after every state change so the app can resume after a
  restart.
