# iOS Project Configuration

Build infrastructure for a Crux iOS shell using XcodeGen, cargo-swift, and
the facet-based codegen binary from the shared crate. Targets Crux 0.17.0-rc3.

## Directory Layout

```
{app-dir}/
    Cargo.toml              # Rust workspace (already exists from core-writer)
    shared/                 # Rust shared crate (already exists)
    iOS/
        project.yml         # XcodeGen project definition
        Makefile            # Build automation
        generated/          # gitignored -- built artifacts
            SharedTypes/    # Swift package from codegen (domain types)
            Shared/         # Swift package from cargo-swift (UniFFI bindings)
        {AppName}/          # Swift source files
            {AppName}App.swift
            Core.swift
            ContentView.swift
            Views/
                LoadingScreen.swift
                MainScreen.swift
                ErrorScreen.swift
```

## XcodeGen `project.yml`

This YAML file defines the Xcode project. Run `xcodegen` to generate the
`.xcodeproj` from it.

### Template

```yaml
name: {AppName}
packages:
  SharedTypes:
    path: ./generated/SharedTypes
  Shared:
    path: ./generated/Shared
  VectisDesign:
    path: ../../../design-system/ios
  Inject:
    url: https://github.com/krzysztofzablocki/Inject.git
    from: "1.5.2"
options:
  bundleIdPrefix: com.vectis.{app-name}
  deploymentTarget:
    iOS: "17.0"
attributes:
  BuildIndependentTargetsInParallel: true
targets:
  {AppName}:
    type: application
    platform: iOS
    sources:
      - {AppName}
    dependencies:
      - package: SharedTypes
      - package: Shared
      - package: VectisDesign
      - package: Inject
    info:
      path: {AppName}/Info.plist
      properties:
        UISupportedInterfaceOrientations:
          - UIInterfaceOrientationPortrait
          - UIInterfaceOrientationLandscapeLeft
          - UIInterfaceOrientationLandscapeRight
        UILaunchScreen: {}
    settings:
      base:
        SWIFT_VERSION: "6.0"
        SWIFT_STRICT_CONCURRENCY: complete
        ENABLE_USER_SCRIPT_SANDBOXING: "NO"
      configs:
        Debug:
          PRODUCT_BUNDLE_IDENTIFIER: com.vectis.{app-name}.debug
          OTHER_LDFLAGS: ["-w", "-Xlinker", "-interposable"]
          EMIT_FRONTEND_COMMAND_LINES: "YES"
        Release:
          PRODUCT_BUNDLE_IDENTIFIER: com.vectis.{app-name}
          OTHER_LDFLAGS: ["-w"]
```

### Adapting the Template

- Replace `{AppName}` with the PascalCase app name (e.g., `TodoApp`).
- Replace `{app-name}` with the kebab-case app name (e.g., `todo-app`).
- The `packages.VectisDesign.path` is relative from the `iOS/` directory to
  `design-system/ios/`. Adjust based on the app's location in the repo.
- No `.udl` files are needed -- Crux 0.17+ uses attribute-based UniFFI
  (`uniffi::setup_scaffolding!()` and `#[uniffi::export]`).
- No `SWIFT_OBJC_BRIDGING_HEADER` -- the Swift packages generated by
  `cargo-swift` handle the FFI bridge internally.
- The `Inject` package enables hot reloading during development. The
  `-Xlinker -interposable` flag and `EMIT_FRONTEND_COMMAND_LINES` setting
  are Debug-only; they are required for InjectionIII to work. In Release
  builds Inject is a no-op stripped by LLVM.

## Makefile

Build automation with three sequential phases: typegen, package, xcode.

### Template

```makefile
.PHONY: all build clean typegen package xcode

SHARED_DIR := ../shared

all: build

# Full build pipeline
build: typegen package xcode

# Phase 1: Generate SharedTypes (domain types as a Swift package)
typegen:
	@echo "Generating SharedTypes..."
	@RUST_LOG=info cargo run --manifest-path $(SHARED_DIR)/Cargo.toml \
		--bin codegen --features codegen,facet_typegen -- \
		--language swift --output-dir generated

# Phase 2: Build Shared (UniFFI bindings as a Swift package via cargo-swift)
package:
	@echo "Building Shared Swift package..."
	@cd $(SHARED_DIR) && \
		cargo swift package --name Shared --platforms ios \
			--lib-type static --features uniffi && \
		rm -rf ../iOS/generated/Shared && \
		mkdir -p ../iOS/generated/Shared && \
		cp -r Shared/* ../iOS/generated/Shared/ && \
		rm -rf Shared

# Phase 3: Generate Xcode project from project.yml
xcode:
	@echo "Generating Xcode project..."
	@xcodegen

# Build the iOS app for Simulator
sim-build:
	@xcodebuild build \
		-project {AppName}.xcodeproj \
		-scheme {AppName} \
		-destination 'platform=iOS Simulator,name=iPhone 16' \
		-configuration Debug \
		CODE_SIGNING_ALLOWED=NO \
		2>&1 | xcbeautify

# Clean build artifacts
clean:
	@rm -rf generated/ *.xcodeproj
```

### Phase Dependency

The three phases must run in order:

1. **typegen** -- runs the codegen binary from the shared crate. It uses
   `TypeRegistry` from facet to extract types at compile time and generates
   a `SharedTypes` Swift package at `generated/SharedTypes/`.
2. **package** -- runs `cargo swift package` in the shared crate directory.
   This builds the static library for iOS targets and wraps it as a
   `Shared` Swift package, then copies it to `generated/Shared/`.
3. **xcode** -- runs `xcodegen` to generate the `.xcodeproj` from
   `project.yml`. The project references both Swift packages.

### Prerequisites

The build requires these tools to be installed:

```bash
cargo install cargo-swift
brew install xcodegen
brew install xcbeautify     # optional, for prettier xcodebuild output
```

## Info.plist

The `Info.plist` is generated by XcodeGen from the `info.properties` in
`project.yml`. For additional entries, add them to the properties map:

```yaml
info:
  path: {AppName}/Info.plist
  properties:
    UILaunchScreen: {}
    UISupportedInterfaceOrientations:
      - UIInterfaceOrientationPortrait
    NSAppTransportSecurity:
      NSAllowsArbitraryLoads: true  # only for development
```

## .gitignore

Add to the app's `.gitignore`:

```
# Xcode
*.xcodeproj/
*.xcworkspace/
DerivedData/
build/

# Generated Swift packages
iOS/generated/

# Build artifacts
iOS/*.xcodeproj
```

Keep `project.yml` and `Makefile` tracked -- these are the source of truth
for the Xcode project configuration.

## Swift Version

Target Swift 6.0 with strict concurrency checking:

```yaml
settings:
  base:
    SWIFT_VERSION: "6.0"
    SWIFT_STRICT_CONCURRENCY: complete
```

## Deployment Target

Minimum iOS 17.0 to ensure availability of:
- `NavigationStack` (iOS 16+)
- `@Observable` macro (iOS 17+)
- Modern SwiftUI APIs (`onChange` with two parameters, etc.)

## Code Signing

For development and CI builds, disable code signing:

```yaml
settings:
  base:
    CODE_SIGNING_ALLOWED: "NO"
```

For distribution builds, configure signing identity and provisioning profile
in the Release configuration.

## Hot Reloading (Inject)

The project includes [Inject](https://github.com/krzysztofzablocki/Inject)
for hot reloading during development. Inject is a no-op in Release builds
(stripped by LLVM), so it can remain in the codebase permanently.

### Build Configuration

The `project.yml` template already includes:

- `Inject` SPM package dependency
- `-Xlinker -interposable` in Debug `OTHER_LDFLAGS`
- `EMIT_FRONTEND_COMMAND_LINES: "YES"` in Debug settings

### Per-Machine Developer Setup

Each developer who wants to use hot reloading must:

1. Download [InjectionIII](https://github.com/nicklama/InjectionIII/releases)
   (or install from the Mac App Store)
2. Place it under `/Applications`
3. Ensure the active Xcode is at `/Applications/Xcode.app`
4. Launch InjectionIII and select the workspace via
   "Open Project / Open Recent"
5. On successful connection the Xcode console shows:
   `💉 InjectionIII connected ...`

### SwiftUI Integration

Every SwiftUI view includes the Inject boilerplate:

```swift
import Inject

struct MyScreen: View {
    @ObserveInjection var inject

    var body: some View {
        // ... view content ...
            .enableInjection()
    }
}
```

- `@ObserveInjection var inject` triggers view reload on injection.
- `.enableInjection()` must be the outermost modifier in the body.
- Both are no-ops in Release builds.
