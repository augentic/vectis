## Design Approach

Findings are grouped into three work streams that can be implemented
independently.

### 1. Data Model Hardening (CRX-003, LOG-003, LOG-005, GEN-005)

- Add `deleted_at: DateTime<Utc>` to `PendingOp::Delete { item_id, deleted_at }`
  for temporal conflict resolution.
- Guard SSE `item_deleted` handler: skip pending ops if `syncing_id` matches
  the item being deleted (current sync in progress).
- Replace `unwrap_or_default()` on state deserialization with `tracing::warn!`
  + explicit error recovery, and surface to the user via an error page.
- Surface `StateSaved(Err(_))` as a retry or user-visible warning instead of
  silently swallowing.

### 2. Operation Coalescing (CRX-004 / LOG-002, LOG-006)

- Before pushing `PendingOp::Delete`, check if the item only exists as a
  pending `Create`. If so, remove the `Create` op and the item from `items`
  without generating a network request.
- Coalesce duplicate `PendingOp::Update` entries for the same item, keeping
  only the latest state.
- Apply same Create-then-Delete elimination for `ClearCompleted`.

### 3. Determinism & Observability (CRX-007, CRX-008, CRX-009)

- **IDs**: Accept an ID from the shell (UUID generated by shell) or request
  one from the Time capability. Remove `next_id` from Model.
- **Timestamps**: Use `crux_time` to obtain `Utc::now()` via a command,
  making the timestamp observable and mockable in tests.
- **ViewModel typing**: Change `active_count: String` to `active_count: usize`
  and `sync_status: String` to `sync_status: SyncStatus` for shell-side
  formatting and localization.
- **Test coverage**: Add edge-case tests for all scenarios listed in CRX-009.

## Sequencing

1. Data Model Hardening (must land first -- changes `PendingOp` shape)
2. Operation Coalescing (depends on new `PendingOp::Delete` shape)
3. Determinism & Observability (independent, but benefits from stable data model)
